Part I: Stitch two key frames

This involves:

  1. compute homography H between two frames;
  2. project each frame onto the same surface;
  3. blend the surfaces.

Check that your homography is correct by plotting four points that form a square in frame 270 and their projections in each image.

Summary of resultst: The images below shows the stich of frame 0270 and 0450. In RANSAC, we randomly pick 4 pair of points each time to solve an exact solution for the linear equation (by np.linalg.solve) and choose the best one in the iteration. Note:

  • Not sure if picking more pairs and solve the least linear square approximately (using numpy.linalg.lstsq or scipy.optimize.least_squares) can be more accurate with a limited time or more efficient.
  • datatype of im and H in cv2.warpPerspective need to be the same (changing them to be both float)
  • For masking, if the image pixels use 0~255 values (rather than [0,1]), remember to convert the array into integer by astyep(np.uint8); otherwise floating point array would be deem as the second case and results in white for those with value larger than 1.0.
  • Key points and their descriptions returned by sift.detectAndCompute can have many duplicate. In the later random selection of 4 points among them, it can happen to get a singular matrix (since we only use 4 points to resort an exact solution rather than more points for an approximate solution.) Hence we eliminate these duplicates before the random selection in RANSAC.

Part II: Panorama using five key frames

In this part you will produce a panorama using five key frames. Let's determine frames [90, 270, 450, 630, 810] as key frames. The goal is to map all the five frames onto the plane corresponding to frame 450 (that we also call the _reference frame_). For the frames 270 and 630 you can follow the instructions in part 1.  

Mapping frame 90 to frame 450 is difficult because they share very little area. Therefore you need to perform a two stage mapping by using frame 270 as a guide. Compute one projection from 90 to 270 and one from 270 to 450 and multiply the two matrices. This produces a projection from 90 to 450 even though these frames have very little area in common.

Summary of results:

  • Directly using the warped/projected frame (with a black background) as the mask is problematic. The main issue is that cv2.warpPerspective can generate non-white, non-black pixel values on the edge of a whole white image. Fortunately this happens only on the boundary (edge) of the image, and we can discard them in the mask by filtering the whole white one (instead of the whole black one!). It's easier when we want to use the added frame part to replace the current frame part; however, if we want to keep the current frame part (as the mask), then we have to simulate the homography transform with the whole white images along the series and use them to obtain the masks.

For each pixel in the sequence of part 3, determine all valid colors (colors that come from all frames that overlap that pixel). Remove camera shake [20 pts]

You can track camera orientation using the homography matrices for each frame. This allows you to estimate and remove camera shake. Please note that camera shake removal for moving cameras is a more difficult problem and is an active area of research in computational photography. One idea (which we haven't tried and might not work) is to assume that camera parameters change smoothly and obtain a temporally smoothed estimate for each camera parameter. A better but more complicated method would be to solve for camera angle and focal length and smooth estimates for those parameters.

Video: https://drive.google.com/file/d/1XQl7FfY_2o9qaEoPgS3VCkKRGLPn3oaz/view?usp=sharing 

Summary: When the time span is not too large, we can assume the camera parameters (mainly the camera angle) change slowly and can be approximate by a linear function. We pick the homography matrix at every interval time (say 2 sec), and calculate the homography matrix between the picked points by linear interpolation. This can essentailly remove the shaking much less than 2 sec, but a trade-off within the 2 sec is on the camera angle changed, which needs to be sufficiently small. The result shows that the shaking is clearly improved from the original video. (However, for freqent shaking happening with duration compared to 2 sec, the improvement is limited.) We also cut the output frame size a bit to remove most black boundary due to the applied homography matrices different from the original ones.

Some more discussion: To insert moving objects, we can do the following.

  1. Insert object points (with parameters of the starting time and moving speed) to the aligned frames. Since the street in the aligned frame is roughly straight.
